## 文字稿

百分之九十五以上的人安装完 Claude Code 以后，第一步就做错了。我把一份能直接复制的 CLAUDE.md 模板整理好了。但你先别急，因为如果你没写好 CLAUDE.md 就开始让它干活，那你后面大概率会遇到内容堆叠、越用越乱，最后你甚至会觉得 Claude 并不好用。但其实问题不在 agent，而在你一开始就没把规则写进去。而这个规则就是 CLAUDE.md。

这个东西不只是一份普通的 readme 文件，它本质上是你和 agent 的顶层协作规则。你先把它写好，它后面才知道该怎么样去跟你配合、什么事情可以直接做、什么事情要先跟你确认、不同的项目该怎么切换。

首先我们先理解最关键的两层。第一层叫做全局 CLAUDE.md，它放在默认目录下，也就是这里。只要你打开 Claude，不管进的是哪个项目，它都会先加载这一层。它解决的是：你是谁？你平时拿它做什么？你做事的原则是什么样的？

第二层是项目级的 CLAUDE.md，它放在每个项目的根目录下，也就是这里。只有当你打开这个项目的时候，它才会被加载。这一层是每个 project 独立的——这个项目目录怎么建、文件怎么命名、有哪些约定需要遵守。前者是管你和 Claude 怎么协作的，而后者是管特定 project 的工作流程的。

就拿我自己来说，我在没写好 CLAUDE.md 的时候，总是会在一个大目录下做很多的小 idea，然后文件就越来越多、结构越来越乱，然后成屎山代码了，然后就只能放弃了。后来呢我就让 Claude 帮我写了一个项目层级的 CLAUDE.md，这样每次进去它就会先按介绍规则来判断这是不是一个新的项目文件夹、怎么样命名合适、结构该怎么展开。写完以后真的体验发生了质变。

然后还有一个很重要的点，就是 CLAUDE.md 和一般的 prompt 或者 skill 不同，它不是越长越好。我实测到了六十到八十行，它就开始遗漏一部分内容。所以我的建议是最好不要超过一百八十行。不要把一些细枝末节写进去，把一些有价值的、跨项目的顶级规则——像是思维准则、行为约束、沟通方式等等——放进去。在写好以后，再使用 Claude Code 的整个体验会稳非常多。
